# AI-администратор стоматологии AVIDENTIKA

Демонстрационный Telegram-бот на Python, который консультирует пациентов только по данным сайта AVIDENTIKA, показывает источник ответа, принимает предварительные заявки и передаёт их администратору. Основная база данных — облачный Supabase/PostgreSQL; SQLite не используется.

> Это демо, а не медицинская система. Бот не ставит диагнозы, не назначает лечение и не подтверждает запись. Окончательные дату и время подтверждает администратор клиники.

## Возможности

- украинский и русский интерфейс с автоматическим определением языка;
- ответы по услугам, ценам, врачам, адресу и графику через RAG;
- pgvector-поиск с резервным полнотекстовым поиском PostgreSQL;
- быстрые кнопки «Записаться на приём» и «Связаться с администратором» под ответами бота;
- кнопка «Задать вопрос» с запросом описать проблему или симптомы;
- безопасная AI-навигация к подходящему направлению и врачу без постановки диагноза;
- честный отказ при отсутствии фактов в базе знаний;
- специальная реакция на признаки потенциально неотложного состояния;
- пошаговая предварительная запись с проверкой украинского телефона;
- обращение к администратору;
- уведомления о новых заявках в `ADMIN_CHAT_ID`;
- кнопки администратора «Взять в работу», «Закрыть заявку» и «Отклонить»;
- автоматические RU/UK-уведомления клиенту при изменении статуса;
- свободные сообщения администратора клиенту через кнопку «Написать клиенту»;
- подтверждение записи с датой, временем, процедурой, врачом и комментарием;
- отдельный статус `confirmed` с сохранением деталей подтверждения в Supabase;
- автоматическое напоминание о подтверждённом визите за 24 часа;
- подтверждение, перенос и отмена визита клиентом с уведомлением администратора;
- команды администратора `/today`, `/tomorrow`, `/new`, `/find`;
- оценка визита от 1 до 5 и текстовый отзыв после закрытия записи;
- хранение статусов `new`, `in_progress`, `closed`, `cancelled`;
- ограничение частоты запросов и длины сообщений;
- защита от prompt injection и утечек внутренних инструкций;
- отдельный безопасный обновляющий скрипт базы знаний;
- Docker и автоматические тесты без обращений к платным API.

## Архитектура

```text
Telegram → handlers → AIService → KnowledgeService → Supabase/pgvector
                    ↘ SupabaseService → заявки и обращения
                    ↘ NotificationService → чат администратора

scripts/update_knowledge.py → сайт AVIDENTIKA → очистка/фрагменты
                            → OpenAI embeddings → Supabase
```

`SUPABASE_SERVICE_ROLE_KEY` используется только серверным Python-процессом. Его нельзя размещать в браузере, на сайте, в мобильном приложении, публичном репозитории или логах.

## 1. Создание Telegram-бота

1. Откройте в Telegram `@BotFather`.
2. Отправьте `/newbot`.
3. Укажите название и username, который заканчивается на `bot`.
4. Скопируйте токен — это значение `TELEGRAM_BOT_TOKEN`.
5. Никому не отправляйте токен и при утечке перевыпустите его через BotFather.

Чтобы узнать `ADMIN_CHAT_ID`:

1. Добавьте созданного бота в администраторский чат или напишите ему лично.
2. Отправьте сообщение.
3. Временно откройте в браузере адрес, заменив весь текст `ВАШ_ТОКЕН` настоящим токеном без `<` и `>`:

```text
https://api.telegram.org/botВАШ_ТОКЕН/getUpdates
```

4. Найдите `message.chat.id`. Для группы значение обычно отрицательное.
5. Если бот работает в группе, дайте ему право отправлять и редактировать свои сообщения.

После получения ID не сохраняйте URL с токеном и очистите историю браузера.

## 2. Создание Supabase

1. Зарегистрируйтесь на Supabase и создайте бесплатный проект.
2. Дождитесь запуска базы.
3. Откройте **Project Settings → API**.
4. Скопируйте Project URL в `SUPABASE_URL`.
5. Скопируйте серверный ключ `service_role` в `SUPABASE_SERVICE_ROLE_KEY`.

Важно: `anon` key не подходит серверному боту с закрытыми RLS-таблицами. `service_role` обходит RLS и даёт полный доступ к данным, поэтому его хранят только в `.env` на сервере.

### Применение SQL-схемы

1. В Supabase откройте **SQL Editor → New query**.
2. Откройте локальный файл `supabase_schema.sql`.
3. Скопируйте весь SQL в редактор.
4. Нажмите **Run**.

Скрипт выполняет `create extension if not exists vector`. Если расширение недоступно автоматически, откройте **Database → Extensions**, найдите `vector`, включите его и повторите SQL.

Схема создаёт таблицы, ограничения, индексы, RLS, HNSW-индекс, триггеры `updated_at` и функции:

- `match_knowledge_documents` — семантический поиск;
- `search_knowledge_documents` — резервный полнотекстовый поиск.

Версия для миграций находится в `supabase/migrations/001_initial_schema.sql`.

Если первоначальная схема уже была применена ранее, дополнительно выполните в SQL Editor файл:

```text
supabase/migrations/002_appointment_confirmation.sql
```

Он безопасно добавляет поля подтверждённой записи и статус `confirmed`, не удаляя существующие заявки.

После этого примените миграцию:

```text
supabase/migrations/003_reminders_reschedule_reviews.sql
```

Она добавляет время визита для планировщика, отметку напоминания, ответы клиента, переносы, оценки и отзывы.

## 3. OpenAI API

1. Создайте API key в кабинете OpenAI Platform.
2. Сохраните его как `OPENAI_API_KEY`.
3. Убедитесь, что у проекта доступен биллинг.

По умолчанию используются:

```dotenv
OPENAI_MODEL=gpt-4.1-mini
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
```

SQL-схема рассчитана на embeddings размерности 1536, которую возвращает `text-embedding-3-small` по умолчанию. При смене embedding-модели сначала согласуйте размерность поля `vector(1536)` и функции поиска.

## 4. Локальная установка

Требуется Python 3.11 или новее.

```bash
git clone https://github.com/baha0011/avidentika-ai-bot.git
cd avidentika-ai-bot
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
cp .env.example .env
```

В Windows активация окружения:

```powershell
.venv\Scripts\activate
```

Если на macOS команда `python3.11` не найдена:

```bash
brew install python@3.11
"$(brew --prefix python@3.11)/bin/python3.11" -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

Заполните `.env`:

```dotenv
TELEGRAM_BOT_TOKEN=токен_от_BotFather
ADMIN_CHAT_ID=числовой_id_чата
OPENAI_API_KEY=секретный_openai_key
OPENAI_MODEL=gpt-4.1-mini
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
SUPABASE_URL=https://PROJECT.supabase.co
SUPABASE_SERVICE_ROLE_KEY=секретный_service_role_key
```

`.env` уже добавлен в `.gitignore`. Не переименовывайте и не коммитьте его.

## 5. Обновление базы знаний

После применения SQL-схемы и заполнения `.env` выполните:

```bash
python scripts/update_knowledge.py
```

Скрипт:

- читает `robots.txt` и учитывает crawl delay;
- начинает с главной страницы и обходит только текущий домен;
- берёт главную, контакты, услуги и страницы врачей;
- удаляет меню, футер, формы и технические элементы;
- сохраняет заголовок, URL, дату и хеш каждого фрагмента;
- обновляет записи без дублей;
- деактивирует старые фрагменты только после успешной загрузки страницы;
- не удаляет рабочие данные при сетевой ошибке;
- сохраняет журнал в `knowledge_update_logs`;
- создаёт локальный отчёт `data/knowledge_update_report.json`.

Не запускайте несколько копий обновления одновременно. Рекомендуемый график после запуска проекта — один раз в сутки или после изменения сайта.

Проверить результат можно в Supabase: **Table Editor → knowledge_documents**. У рабочих строк `is_active = true`, а `embedding` заполнен.

## 6. Запуск

```bash
python bot.py
```

При верной конфигурации бот запустит long polling. Для остановки нажмите `Ctrl+C`.

Если Telegram отвечает `409 Conflict` из-за старого webhook, один раз откройте в браузере следующий адрес, заменив `ВАШ_ТОКЕН` настоящим токеном без скобок:

```text
https://api.telegram.org/botВАШ_ТОКЕН/deleteWebhook
```

Команды Telegram:

- `/start` — главное меню;
- `/help` — краткая помощь;
- `/cancel` — отмена текущего сценария;
- `/privacy` — сведения о данных и ограничениях демо.

Команды администратора:

- `/today` — подтверждённые записи на сегодня;
- `/tomorrow` — записи на завтра;
- `/new` — новые заявки и обращения;
- `/find A-1234ABCD` — поиск по номеру заявки.

## 7. Запуск через Docker

```bash
docker compose build
docker compose up -d
docker compose logs -f bot
```

Остановка:

```bash
docker compose down
```

Секреты передаются через локальный `.env` и не копируются в образ. Контейнер работает от непривилегированного пользователя, корректно получает `SIGTERM` и перезапускается после сбоя.

Обновление базы знаний из одноразового контейнера:

```bash
docker compose run --rm bot python scripts/update_knowledge.py
```

## 8. Проверка записи

1. Отправьте боту `/start`.
2. Нажмите «Записаться на приём».
3. Введите имя.
4. Передайте телефон кнопкой Telegram или в формате `+380XXXXXXXXX` / `0XXXXXXXXX`.
5. Укажите услугу, желаемую дату, время и комментарий; необязательные поля можно пропустить.
6. Проверьте итог и нажмите «Подтвердить».
7. Убедитесь, что бот сообщил: заявка предварительная.
8. Откройте **Supabase → Table Editor → appointments**: должна появиться строка со статусом `new`.
9. В администраторском чате нажмите «Взять в работу».
10. Проверьте статус `in_progress`, ID администратора, уведомление клиента и обновлённое сообщение.
11. Нажмите «Подтвердить запись» и введите дату, время, процедуру, врача и комментарий.
12. Проверьте статус `confirmed` и сообщение с деталями, полученное клиентом.
13. Кнопка «Написать клиенту» позволяет отправить клиенту произвольное сообщение администратора.
14. После приёма нажмите «Закрыть заявку» и проверьте `closed_at`.

Аналогично протестируйте «Связаться с администратором» и таблицу `support_requests`.

## 10. Логи и безопасность

Логи содержат запуск, обновление базы знаний, номера заявок, смену статусов и типы ошибок. Полные телефоны, API-ключи, тексты медицинских обращений и `service_role` не логируются. Пользователь не получает traceback.

Дополнительные меры:

- callback управления заявкой работает только в `ADMIN_CHAT_ID`;
- входные данные ограничены по длине;
- HTML экранируется перед отправкой в Telegram;
- OpenAI получает только несколько релевантных фрагментов, а не весь сайт;
- инструкции из документов считаются данными и не исполняются;
- RLS закрывает таблицы для `anon` и `authenticated`;
- HTTP имеет таймауты и повторные попытки временных ошибок.

## 12. Подключение CRM и реального расписания

Для CRM добавьте отдельный адаптер в `app/services`, вызываемый после успешного сохранения в Supabase. Передавайте `public_id` как idempotency key, чтобы повторный запрос не создал дубль. Статусы CRM следует синхронизировать webhook-обработчиком с проверкой подписи.

Для реального расписания нужен официальный API клиники или CRM:

1. получать доступные интервалы из API;
2. временно блокировать выбранный слот;
3. создавать запись идемпотентно;
4. подтверждать её только после успешного ответа API;
5. обрабатывать отмену и перенос;
6. учитывать часовой пояс `Europe/Kyiv`.

До такой интеграции бот намеренно принимает только предварительную заявку и не сообщает, что время забронировано.

## Ограничения демо

- содержание зависит от успешного запуска `update_knowledge.py` и актуальности сайта;
- без API расписания нельзя показать реальные свободные окна;
- AI и embeddings требуют доступного OpenAI API и оплачиваются по тарифам аккаунта;
- при недоступности pgvector включается полнотекстовый поиск, который хуже понимает перефразирования;
- автоматическое определение языка для очень коротких нейтральных сообщений может выбрать украинский — это безопасный язык по умолчанию;
- демо не заменяет медицинскую консультацию и требует проверки владельцем клиники перед промышленным использованием.

## Частые ошибки

- `401 Unauthorized` Telegram — проверьте `TELEGRAM_BOT_TOKEN`, отсутствие пробелов и скобок `< >`.
- `409 Conflict` Telegram — удалите активный webhook перед запуском polling, как показано выше.
- `429 Too Many Requests` OpenAI — исчерпана квота или не настроен биллинг; это не ошибка Telegram-токена.
- Ошибка сборки `cryptography` на старой среде — используйте Python 3.11 и предварительно обновите `pip`.
- Ошибка Supabase — проверьте `SUPABASE_URL`, именно серверный `SUPABASE_SERVICE_ROLE_KEY` и применённую SQL-схему.
